Create `CONTENT_CONVENTIONS.md` in the project root. This document defines the structure for each question type so that Phase 5 content generation follows consistent patterns. Include:

1. **Header**: Title "Content Conventions" with purpose statement: reference for content authors (human or AI) when creating questions.

2. **Frontmatter Schema** section: Show the complete frontmatter template with all fields:
```yaml
---
ua_question: "..."
en_question: "..."
ua_answer: |
  ...
en_answer: |
  ...
section: "section-slug"
order: N
tags: [tag1, tag2]
type: "basic" | "deep" | "trick" | "practical"
---
```
Note: `type` defaults to `basic` if omitted. Existing v1.0 questions omit it.

3. **Question Types** section with subsections for each type:

**Basic** (default): Standard interview questions. Answer structure: direct explanation (2-4 paragraphs), optionally with code example. No special callouts.

**Deep Technical** (`type: deep`): Questions probing deep understanding of internals, edge cases, or architecture. Answer structure: start with the core concept, then dive into implementation details, include code showing internal mechanism or advanced usage, end with "why it matters" in interviews.

**Trick Questions** (`type: trick`): Questions with a common wrong answer or subtle gotcha. Answer structure: MUST start with a `> **Trap:** {the common misconception}` blockquote callout, then explain the correct answer, include code demonstrating the trap vs correct behavior.

**Practical Scenarios** (`type: practical`): Real-world problem-solving questions. Answer structure: begin with `**Scenario:**` describing a realistic situation, follow with `**Approach:**` outlining the solution strategy, end with `**Solution:**` including concrete code or configuration.

4. **File Naming** section: Files use kebab-case slug: `src/content/questions/{section}/{slug}.md`. Slug should reflect the question topic.

5. **Order Field** section: Existing v1.0 questions use orders 1-15 per section. New advanced questions (Phase 5) should start at order 16 to avoid collisions. Within a section, use: 16-20 for deep, 21-25 for trick, 26-30 for practical.

6. **Tags** section: Tags are lowercase, hyphenated strings describing the topic. Each question should have 2-4 tags. Use existing section-specific tag conventions (e.g., Java: `concurrency`, `collections`, `jvm`; Docker: `networking`, `volumes`, `security`).

7. **Bilingual Content Rules**: Both UA and EN versions are mandatory. Answers should be substantive (3-6 paragraphs with code). Code examples are identical in both languages. Explanatory text is translated, not transliterated.
